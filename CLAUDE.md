## 一、项目背景

- **项目名**：制片帽（Producer Hat）
- **定位**：面向影视制片人的 AI 工具，核心目的是**提高制片工作效率**。
- **目标用户**：影视行业从业者 / 制片人（先给帽帽自己用，未来面向广大从业者）。
- **核心功能规划（三大板块）**：
  | 板块 | 做什么 | 状态 |
  |---|---|---|
  | **策划 · 选题雷达** | 在大买家之前，从各大文学网站发现还没卖、有改编潜力的好故事，给出改编分析报告 | 已上线（策划 context 维护） |
  | **制作 · 制片统筹** | 上传剧本 → 解剖分析 + 分场/顺场表 + 参考预算，全可编辑（详见【九】） | **v1 已上线（制作 context 维护）** |
  | **发行** | 数据监测、档期竞品、宣发节点等 | 待定 |
- **平台节奏**：先做电脑端（网页），成熟后上手机端（APK）。

## 二、技术栈（按当前代码实际情况，非臆测）

| 层 | 用了什么 | 大白话 |
|---|---|---|
| 前端 | **原生 HTML / CSS / JavaScript**，无框架、无构建、无第三方 JS 库 | 就是最朴素的网页三件套。没用 React/Vue 这类"脚手架"。一个单页应用，靠 JS 切换"屏幕"。 |
| 前端状态/存储 | **浏览器 localStorage** | 账号、口味偏好、收藏、点赞都存在用户自己浏览器里，没上传。 |
| 后端 | **Python 3.14.3 标准库**（`http.server` + `urllib`），**零第三方依赖** | 一个脚本就是后端，不用 pip 装任何东西，谁的 Mac 都能跑。 |
| 数据库 | **没有数据库**。数据放在 JSON / JS 文件里 | 候选作品库 = `backend/corpus.json`；线上静态演示数据 = `app/data/recommendations.js`。 |
| 大模型 | **DeepSeek API**：模型 `deepseek-v4-pro`（思考型、最强；旧名 `deepseek-chat` 将于 2026/07/24 弃用），接口 `https://api.deepseek.com/chat/completions`（OpenAI 兼容格式） | 大脑用国产 DeepSeek。key 放本地 `backend/key.txt`，不进代码/git。单次约 80 秒（思考型偏慢），按周用没问题。 |
| 字体 | 造字工房 **新简黑 / 骏雅 / 书研**（.otf，本地打包，共约 5.7MB） | 文艺调性。⚠️ 见【已知问题】授权事项。 |
| 部署 | **GitHub Pages**（静态前端，push 自动部署）+ **本地 Python 后端**（AI 功能） | 线上：https://xmoses909-debug.github.io/zhipianmao/ ；仓库：github.com/xmoses909-debug/zhipianmao |

> **重要**：AI 实时选片只在**本地后端**可用；**线上（Pages）是纯静态版**，AI 按钮会优雅提示"需本地后端"。原因：静态网页不能安全地直接调大模型（密钥会暴露）。

## 四、已做的架构决策（每条带「为什么」+「放弃了什么」）

| 决策 | 为什么这么选 | 放弃 / 不选 |
|---|---|---|
| **前端纯原生、无框架、无构建** | 简单、零依赖、非技术用户能双击就开、易托管；项目目前不复杂，框架是负担 | 放弃 React/Vue（徒增复杂度与构建步骤） |
| **后端用 Python 标准库、零依赖** | 用户 Mac 自带 python3，不用装任何东西就能跑 | 放弃 Node/Flask/FastAPI（都要装依赖、抬高门槛） |
| **不用数据库，数据放 JSON/JS 文件** | 测试阶段够用、零运维 | 放弃数据库（真多用户上线时才需要，见 TODO） |
| **大模型走国产 DeepSeek，且设计成可切换** | 面向国内、国内可访问、便宜、合规；`deepseek-v4-pro`（思考型）读小说+文学判断强；想省钱可一行切 `deepseek-v4-flash` | 放弃 Claude API（不正式对华、访问与付费麻烦）；暂不买 MiniMax 包年（测试期按量更划算） |
| **选品评分：好故事 > 题材**（综合分 = 故事力×0.6 + 题材×0.4） | 帽帽要的是大模型的**文学判断力**，不止标签匹配；宁可推题材不全对但故事极好的 | 放弃"纯标签匹配" |
| **选品策略：选品前置**（找未售的早期作品，不追热榜） | 热门 IP 早被头部公司买空，推了也没用 | 放弃"追热度榜" |
| **候选库锚定真实事实 + 大模型只做判断** | 书名/作者/链接/字数/梗概来自真实库，**不让大模型编造书目**（对制片人是硬伤）；判断/分析才交给大模型 | 放弃"让大模型自由生成书单"（会幻觉假书） |
| **三源自动抓，豆瓣为主力**（帽帽明确要求 2026-06-08） | 帽帽要豆瓣的小众原创连载。豆瓣 robots 禁 `/j/`、有封查风险——**帽帽拍板接受**；缓解：真浏览器 UA + 1 小时缓存 + 不暴力并发。发现入口 `/j/tag/<标签>`（从豆瓣 JS 源码里挖出来的，无需 cookie、可翻页） | 放弃"只精选收录豆瓣"（那样豆瓣进不了池）；**商用前仍需谈授权**（robots 能爬 ≠ 商用授权） |
| **三源按配额组池**（豆瓣14+晋江6+番茄4，豆瓣为主） | 晋江一次 400 本会把别源挤光；配额保证三源都进大模型视野，且豆瓣占主力 | 放弃"纯按分数混排"（会变单一源） |
| **标签按 4 类归组**（时代/类型/题材/调性） | 原来一堆扁平标签太乱；归类后用户好选、前后端词表对齐 | 放弃扁平标签；老用户旧词自动迁移（日系文艺→文艺等） |
| **耽美/ABO 不作过审门槛** | 帽帽口径：过审是剧本层改人物关系的事，不该把好故事滤掉 | 放弃"按过审先筛掉耽美" |
| **账号 = 本地演示版（localStorage）** | 静态托管做不到真账号 | 真多用户账号需后端 + 数据库（见 TODO） |
| **部署 GitHub Pages** | 免费、push 自动部署、能很快上线让人用 | ⚠️ 这是**临时**选择：github.io 国内访问不稳，面向大众时要换国内托管 |
| **小范围试用走 Cloudflare 免费隧道**（`分享给朋友.command`） | 给几个朋友试不值得为此备案/租服务器；隧道免费、免备案、即开即用，且 **key 不出本机**（最安全） | 放弃"为试用就上国内云"（备案数天~数周+成本）；代价=本机要开着、网址每次变、国内偶尔慢 |
| **后端能认 `PORT` 环境变量** | 让 Claude 预览工具/隧道能分配端口，不写死 4173 | —— |
| **Service Worker 暂停** | 反复编辑时旧缓存老干扰；先求"改了就能看到" | 部署稳定后再启用稳健版 SW |
| **key 走本地 `key.txt`/`.env`（git 忽略）** | 密钥绝不进代码、git、聊天 | 放弃硬编码、放弃前端直连大模型 |
| **字体造字工房**（标题骏雅、正文骏雅、点缀书研） | 文艺调性；帽帽有授权。标题原用新简黑但太粗"坨在一起"，改骏雅 | 放弃新简黑做标题；⚠️ 商用授权见 TODO |
| **openclaw 不作产品后端** | 它是单用户、聊天驱动的私人助理，不适合做多用户网页后端；用专用轻量后端反而模型可切换、好维护 | 放弃把 openclaw 当产品后端（可作帽帽个人助理或参考） |

## 九、制作板块（制片统筹系统）—— 2026-06-10 上线 v1

> **本章由"制作 context"维护**；策划 context 改共享文件前请先读本章的互不干扰约定。

### 9.1 功能（三个区域，全部可编辑）
1. **🔬 剧本解剖**：上传剧本 → `deepseek-v4-pro` 通读全本出报告：一句话故事 / 梗概 / 主题 / 类型调性 / 结构节拍 / 人物小传+选角建议 / 亮点 / 风险（过审·制作·市场）/ 体量评估 / 制片总评。**超 6 万字取头 70%+尾 30% 采样并如实注明**。
2. **🎬 分场表 + 顺场表**：正则识别场头（`1、内 X 日`/`第N场`/`INT.`/`3-12` 等 6 种模式，谁命中多用谁；都识别不出就按字数分块让 AI 自行断场并提示核对）→ `deepseek-v4-flash` 分批提取（每批 ~4500 字）。**字段对齐帽帽给的《开盘》场景表行业模板**：集/场/气氛(日夜晨昏)/内外/**主场景+分场景两级**/页数/内容提示/动作特效/主要演员/**特约演员/群众演员分列**/服化道提示。**顺场表 = 分场表按"主场景"归组的前端实时视图**，改分场表自动跟着变。单元格直接点开改、可增删行；**CSV 导出为演员矩阵格式**（每个演员一列、出场打 ●，同《开盘》，排档期用；带 BOM Excel 直接开）。
3. **💰 参考预算**：按中国市场行情（2024-2026），结合分场表统计 + 项目定位（形态/级别/集数）→ `deepseek-v4-pro` 出科目分组预算（low/high 区间、万元）+ 拍摄周期 + 前提假设 + 行情参考 + 免责。行级可编辑、增删行、合计实时重算、CSV 导出。prompt 里明确"拿不准别编具体片名数字"防幻觉。

**上传格式（v1.5）**：.txt / .docx / .pdf / .fdx / **.zip**，≤60MB。docx 用 zipfile+xml 零依赖解析；GBK 自动识别；**PDF 用自写的 `pdftext.py` 纯标准库提取**（FlateDecode+ObjStm+ToUnicode CMap，支持 Word/WPS/浏览器导出的文本型 PDF；扫描件/加密件报中文错误引导转格式；NFKC+部首映射修字体子集的"⽇≠日"坑）。
**多集剧（v1.5）**：ZIP 内多个文件 → 每文件一集；单文件含"第N集"标记 → 自动按集切分（页眉重复集号有抗干扰）。生成**剧集组**：一集一个项目（可单集解剖/拆场/预算），剧集页有"依次拆全部集"（断点续跑：已拆的集自动跳过）+ **总分场表/总顺场表**（聚合各集实时推导，CSV 演员矩阵导出）。ZIP 老压缩工具的 GBK 文件名自动修复，"第2集"排在"第10集"前（自然排序）。

- `POST /api/production/run {kind}` → 返回 jobId → 后台线程跑 → 前端 2.5s 轮询 `GET /api/production/job?id=`。
